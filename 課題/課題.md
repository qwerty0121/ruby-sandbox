## 第1問：シンボルとハッシュによる簡易受発注システム


### 1. 課題タイトルと概要

- 商品の注文情報を受け取り、合計金額を計算するプログラムの作成

### 2. 実装してほしい仕様（要件）

- 商品名（シンボル）と単価（数値）を格納した「在庫ハッシュ」をあらかじめ定義する。
- `[ { item: :apple, quantity: 3 }, { item: :banana, quantity: 2 } ]` のような注文データ（ハッシュの配列）を受け取り、合計金額を返すメソッドを実装する。
- 在庫ハッシュに存在しない商品が指定された場合は、`0`（または例外ではなく特定のエラーハンドリング）として処理する。

### 3. 意識してほしい概念

- シンボル、ハッシュの操作、配列の反復処理（`each` などの基本操作）

### 4. テストケース

- マスタ: `{ apple: 100, banana: 80, orange: 120 }`
- ケース1：
    - 入力: `[{ item: :apple, quantity: 2 }, { item: :orange, quantity: 1 }]`
    - 出力: `320`
- ケース2：
    - 入力: `[{ item: :banana, quantity: 5 }]`
    - 出力: `400`
- ケース3（未登録商品を含む）：
    - 入力: `[{ item: :apple, quantity: 1 }, { item: :grape, quantity: 3 }]`
    - 出力: `100`

- 数値のメソッド（`%` 演算子など）、`if` / `unless` 修飾子、論理演算子

### 4. テストケース

- ケース1：
    - 入力: `2020`
    - 出力: `true`
- ケース2：
    - 入力: `1900`
    - 出力: `false`
- ケース3：
    - 入力: `2000`
    - 出力: `true`

- ユーザーデータ（ハッシュの配列）から、特定の条件を満たすユーザーの特定属性を加工・抽出する

### 2. 実装してほしい仕様（要件）

- `[ { name: "Alice", age: 25, role: :admin }, ... ]` のような配列を受け取る。
- 「年齢が20歳以上」かつ「roleが :admin ではない」ユーザーの「名前（`name`）」を、すべて大文字（UPPERCASE）に変換した配列として抽出する。
- `for` や `each` を使わず、`Enumerable` モジュールの高階関数（メソッドチェーン）だけでエレガントに記述すること。

### 4. テストケース

    ```ruby
    users = [
      { name: "Alice", age: 25, role: :admin },
      { name: "bob", age: 19, role: :member },
      { name: "Charlie", age: 30, role: :member },
      { name: "dave", age: 22, role: :guest }
    ]
    ```
    
- ケース1（上記のデータ全体を投入）：
    - 出力: `["CHARLIE", "DAVE"]`

    - 入力: `[{ name: "emily", age: 20, role: :member }]`

## 第5問：破壊的メソッドと非破壊的メソッドの挙動理解


### 1. 課題タイトルと概要

- 配列に含まれる文字列の末尾から、特定の改行コードや不要な空白を除去し、重複を排除した一意なリストを作成する

### 2. 実装してほしい仕様（要件）

- 文字列の配列（要素の末尾に空白や改行を含む可能性がある）を受け取る。
- 各要素の「前後の空白・改行を取り除いた状態」にし、さらに「重複を排除」した配列を返す。
- この処理を行う際、引数として渡された**元の配列自体を書き換える（破壊する）メソッド**と、**元データを汚さない（非破壊的な）メソッド**の2パターンをそれぞれ別メソッドとして実装し、オブジェクトの識別子（`object_id`）が変わるか・変わらないかを確認できるようにする。

### 3. 意識してほしい概念

- `String#strip`, `Array#uniq`（それぞれの破壊的バージョン `!` との違い）、オブジェクトの参照と複製

### 4. テストケース

- 入力データ: `arr = [" ruby \\n", "java", " ruby", "python "]`
- ケース1（非破壊メソッドを呼び出した場合）：
    - 出力: `["ruby", "java", "python"]`
    - 検証: 元の変数 `arr` の中身が変わっていないこと。
- ケース2（破壊的メソッドを呼び出した場合）：
    - 出力: `["ruby", "java", "python"]`
    - 検証: 元の変数 `arr` の中身自体が `["ruby", "java", "python"]`（または途中の破壊状態）に変形していること。

## 第6問：独自クラスの定義と `initialize` によるカプセル化


### 1. 課題タイトルと概要

- 銀行口座を表す `BankAccount` クラスの作成

### 2. 実装してほしい仕様（要件）

- 初期化時（`initialize`）に「口座名義（文字列）」と「初期残高（数値）」を受け取る。
- `deposit(amount)` メソッドで入金、`withdraw(amount)` メソッドで出金ができる。
- 残高不足の状態で出金しようとした場合は、残高は減らさずに `"Shortage"` という文字列を返す（例外は使わない）。
- 口座名義は外部から変更できない（読み取り専用）が、残高は外部から直接書き換えられず、メソッドを通じてのみ変更できるようにカプセル化（アクセササフィックスの適切な設定）を行う。

### 3. 意識してほしい概念

- クラス定義、`initialize`、`attr_reader`、インスタンス変数（`@`）

### 4. テストケース

- ケース1：
    - 操作: `account = BankAccount.new("Sato", 1000); account.deposit(500); account.balance`
    - 出力: `1500`
- ケース2（残高不足）：
    - 操作: `account = BankAccount.new("Sato", 1000); account.withdraw(1500)`
    - 出力: `"Shortage"`（かつ `account.balance` は `1000` のまま）
- ケース3（名義の変更不可チェック）：
    - 操作: `account = BankAccount.new("Sato", 1000); account.name = "Tanaka"`
    - 出力: `NoMethodError` が発生すること（変更のためのセッターメソッドが存在しない）

## 第7問：ブロックを受け取る独自メソッドの作成（制御構造の自作）


### 1. 課題タイトルと概要

- 与えられた回数だけ処理を繰り返し、その時のインデックスをブロックに渡す、簡易版 `.times` メソッドの自作

### 2. 実装してほしい仕様（要件）

- `repeat_process(n)` というメソッド（または任意のクラスメソッド）を定義する。
- 引数 `n` 回だけ、メソッドに渡されたブロックを実行する。
- ブロックには、現在のループカウンタ（`0` から `n-1` まで）を引数として渡す（`yield` を使用）。
- もしブロックが渡されなかった場合は、`"No block given"` という文字列を返すか、Ruby標準の挙動（Enumeratorを返すなどではなく今回はメッセージでよい）にする。

### 3. 意識してほしい概念

- `block_given?`、`yield`、ブロック引数の受け渡し

### 4. テストケース

- ケース1：
    - 入力: `repeat_process(3) { |i| puts i }`
    - 出力: （標準出力に）`0`、`1`、`2` が順に出力される。
- ケース2（ブロック内での集計）：
    - 入力: `sum = 0; repeat_process(5) { |i| sum += i }; sum`
    - 出力: `10`
- ケース3（ブロックなし）：
    - 入力: `repeat_process(5)`
    - 出力: `"No block given"`

## 第8問：例外処理（内部例外の捕捉と再発生）


### 1. 課題タイトルと概要

- 文字列を整数に変換して割り算を行う計算機メソッドにおける例外ハンドリング

### 2. 実装してほしい仕様（要件）

- `safe_divide(str_a, str_b)` メソッドを作成する。
- 引数は両方とも文字列。これらを整数に変換（`.to_i` ではなく、不正な文字列で例外が発生する `Integer(str)` を使用すること）して、`a / b` の計算を行う。
- 発生し得る2つの例外（引数が数字ではない場合の `ArgumentError`、`0` で割ろうとした場合の `ZeroDivisionError`）を個別に捕捉（`rescue`）する。
- `ZeroDivisionError` の場合は、一律で `Float::INFINITY`（または特定の記号/数値）を返す。
- `ArgumentError` の場合は、独自のメッセージ `"Invalid Number"` を持たせた新たな例外（RuntimeErrorなど）を発生（`raise`）させる。

### 3. 意識してほしい概念

- `begin-rescue-end`、複数の `rescue` 節、`raise`、組み込みの例外クラス

### 4. テストケース

- ケース1（正常系）：
    - 入力: `safe_divide("10", "2")`
    - 出力: `5`
- ケース2（ゼロ除算）：
    - 入力: `safe_divide("10", "0")`
    - 出力: `Infinity` （`Float::INFINITY`）
- ケース3（不正な入力）：
    - 入力: `safe_divide("10", "xyz")`
    - 出力: `RuntimeError` が発生し、メッセージが `"Invalid Number"` であること。

## 第9問：オブジェクト指向：クラスの継承とメソッドオーバーライド


### 1. 課題タイトルと概要

- 会員システムにおける「通常会員」と「プレミアム会員」の料金計算モデルの構築

### 2. 実装してほしい仕様（要件）

- 基底クラス `RegularMember` を作成する。属性に `name`（名前）を持つ。
- `charge(base_price)` メソッドを持ち、通常会員は引数の `base_price`（基本料金）をそのまま返す。
- `RegularMember` を継承した `PremiumMember` クラスを作成する。
- `PremiumMember` の `charge(base_price)` メソッドは、基本料金を「2割引（`base_price * 0.8`）」にした上で、プレミアム会員の一律固定手数料 `500`（円）を加算した金額を返すようにオーバーライドする。
- クラスの継承関係が正しく成り立っていることを意識する。

### 3. 意識してほしい概念

- クラスの継承（`<`）、メソッドオーバーライド、オブジェクト指向の多態性（ポリモーフィズム）

### 4. テストケース

- ケース1（通常会員）：
    - 操作: `member = RegularMember.new("Alice"); member.charge(3000)`
    - 出力: `3000`
- ケース2（プレミアム会員）：
    - 操作: `premium = PremiumMember.new("Bob"); premium.charge(3000)`
    - 出力: `2900` （`3000 * 0.8 + 500`）
- ケース3（継承関係の確認）：
    - 操作: `PremiumMember.new("Bob").is_a?(RegularMember)`
    - 出力: `true`

## 第10問：Mix-in（モジュールの組み込み）による共通機能の拡張


### 1. 課題タイトルと概要

- 異なるクラス（商品クラスとログクラス）に対して、共通の「JSON風文字列出力機能」をモジュールで提供する

### 2. 実装してほしい仕様（要件）

- `Serializable` モジュールを作成する。
- このモジュールの中に `to_json_string` メソッドを定義する。
- `to_json_string` は、そのクラスが持っている「インスタンス変数」の名前と値を解析し、`"{"@name":"Apple","@price":100}"` のような文字列を動的に生成して返す（※本物のJSONライブラリは使わず、Rubyのインスタンス変数リフレクションメソッドを使うこと）。
- `Product` クラスと `Log` クラスを定義し、それぞれに `Serializable` モジュールを `include`（Mix-in）して、どちらのインスタンスからも `to_json_string` が呼び出せることを確認する。

### 3. 意識してほしい概念

- `Module` 定義、`include`、`instance_variables`、`instance_variable_get`

### 4. テストケース

- ケース1（Productクラスでの利用）：
    - 定義: `Product` は `@name`, `@price` を初期化で持つとする。
    - 操作: `p = Product.new("Apple", 100); p.to_json_string`
    - 出力: `"{\\"@name\\":\\"Apple\\",\\"@price\\":100}"` （キーの順序は不問）
- ケース2（Logクラスでの利用）：
    - 定義: `Log` は `@msg`, `@level` を初期化で持つとする。
    - 操作: `l = Log.new("Login success", "INFO"); l.to_json_string`
    - 出力: `"{\\"@msg\\":\\"Login success\\",\\"@level\\":\\"INFO\\"}"`
- ケース3（Mix-inの確認）：
    - 操作: `Product.include?(Serializable)`
    - 出力: `true`
